# 多Agent工作流和流式输出测试说明

## 测试概述

本文档说明如何测试多Agent工作流和流式输出功能。

## 测试内容

### 1. PhysicsAssistantAgent 测试

测试 `PhysicsAssistantAgent` 的基本功能，包括：
- RAG检索
- LLM生成回复
- 流式输出
- 非流式输出
- 对话历史支持

### 2. 流式输出测试

测试流式输出的实时性和完整性：
- 验证chunk的实时接收
- 验证完整响应的拼接
- 验证来源和推荐资源的返回

### 3. 对话历史测试

测试带对话历史的Agent执行：
- 验证对话历史的正确传递
- 验证上下文理解能力

## 运行测试

### 前置条件

1. 确保后端服务正在运行
2. 确保Ollama服务可用
3. 确保MongoDB数据库连接正常
4. 确保RAG服务可用（如果测试RAG功能）

### 运行测试脚本

```bash
cd sensor-api
python test_agent_workflow.py
```

### 测试输出说明

测试脚本会输出：
1. **基本流式输出测试**: 测试基本的流式输出功能
2. **带对话历史的流式输出测试**: 测试带对话历史的流式输出
3. **非流式输出测试**: 测试非流式输出功能

每个测试会显示：
- 实时流式输出内容
- 总chunk数
- 总响应长度
- 来源信息
- 推荐资源

## 故障排查

### 问题1: 流式输出中断

**可能原因**:
- 网络连接不稳定
- Ollama服务响应超时
- 后端服务异常

**解决方法**:
- 检查网络连接
- 检查Ollama服务状态
- 查看后端日志

### 问题2: HTML渲染失败

**可能原因**:
- DOMPurify未正确加载
- HTML内容格式不正确
- CSS样式未正确应用

**解决方法**:
- 检查浏览器控制台错误
- 验证HTML内容格式
- 检查CSS样式文件

### 问题3: RAG检索失败

**可能原因**:
- 向量数据库连接失败
- 检索服务不可用
- 查询参数错误

**解决方法**:
- 检查向量数据库连接
- 检查检索服务状态
- 验证查询参数

## 性能测试

### 流式输出延迟测试

测试从请求发送到第一个chunk接收的延迟：
- 目标: < 2秒
- 测试方法: 记录请求时间和第一个chunk接收时间

### 完整响应时间测试

测试完整响应的生成时间：
- 目标: 根据响应长度而定
- 测试方法: 记录请求时间和完成时间

## 注意事项

1. **测试环境**: 确保在测试环境中运行，避免影响生产环境
2. **资源消耗**: 流式输出测试会消耗较多资源，注意监控
3. **超时设置**: 大模型生成可能需要较长时间，适当设置超时时间
4. **日志记录**: 测试过程中注意查看日志，便于问题排查
